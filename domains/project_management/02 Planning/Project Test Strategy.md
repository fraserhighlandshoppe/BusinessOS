# Project Test Strategy

## Inputs to the Strategy
- Agreements
- Business Requirements
- Project Charter
- Project Quality Plan
- Risk Management Documents

## Steps in Developing Test Strategy

### 1. Prepares to Define Test Strategy
Review applicable work products to identify project needs.

**Relevant Work Products**:
- Agreement with sponsors/client
- Business requirements specification
- Business roles and locations
- Change cases
- Current solution evaluation
- Project charter
- Project quality plan
- Release plan
- Risk management documentation
- Service request
- System requirements specification
- System/component architecture
- User group
- User profiles

**Note**: Smaller projects may develop simple test strategy recorded in master test plan.

### 2. Reviews Existing Test Assets
- Reviews available test assets for reuse
- When enhancing existing systems, strategy may already exist
- Search IC repositories for suitable documents
- Submit success to Worldwide Asset Reuse Reporting Database if saved cost/time

### 3. Determines Need for Test Strategy
- Less complex projects may choose not to create separate test strategy
- Details can be incorporated into master test plan

### 4. Develops and Documents Test Strategy
- Includes high-level description of "what" to be tested
- Master test plan includes detail-level descriptions

**Development Approach**:
- Consultative involving all affected groups:
  - Client and end users
  - Client test teams
  - Development and internal test teams
  - Managed operations group
  - Business analysts and system designers
  - Any other group with assigned responsibility

**Best Practice**: Workshop environment for consultative development.

### 5. Validates Test Strategy
Possible validation techniques:
- Early usability evaluation
- User interface prototype
- System test architecture
- Requirements traceability and verification matrix
